# Lodia 最小 MVP 实施计划

版本：v0.1  
日期：2026-05-05  
关联文档：[LODIA_PRD.md](../LODIA_PRD.md)、[LODIA_TECH_ARCHITECTURE.md](../LODIA_TECH_ARCHITECTURE.md)、[LODIA_TECH_RISK_ASSESSMENT.md](../LODIA_TECH_RISK_ASSESSMENT.md)  
目标：用最小工程量跑通“Case 提交 -> 脱敏标注 -> 人工审核 -> 数据集导出 -> 使用账本 -> 模拟收益”的完整闭环。

## 1. MVP 定义

Lodia MVP 不做完整数据市场，也不做真实打款。第一版只验证一个问题：

> 一个普通用户能否把 AI 工作记录提交给 Lodia，并在平台内看到它被脱敏、结构化、审核、打包成数据集，并产生可追踪的使用记录和收益凭证？

如果这个闭环跑通，Lodia 的核心价值就成立：个人 AI 工作记录可以被转化为可治理、可授权、可复用、可分账的数据资产。

## 2. 最小闭环范围

### 2.1 必须包含

- 贡献者账号和开发态登录。
- 文本/Markdown/JSON/TXT 文件上传。
- 邮件转发模拟入口：用“粘贴邮件内容/上传 `.eml` 文本”的方式替代真实收件服务。
- Raw Quarantine 原始隔离区。
- 自动脱敏：手机号、邮箱、姓名占位、身份证样式、地址样式、API Key、token、URL query、客户名/公司名粗略识别。
- 二次残留扫描。
- Case schema 标准化。
- raw_hash、canonical_hash、SimHash/Jaccard 轻量去重。
- 自动标签：领域、任务类型、难度、可复用用途、质量分、风险分。
- Quality Gate 和 DRL0-DRL3。
- 人工审核工作台：通过、拒绝、改标签、升级/降级 DRL。
- Dataset Builder：从 DRL3 Case 生成数据集。
- Dataset Manifest、Quality Report、JSONL/CSV/Markdown 导出。
- Usage Ledger：记录 `dataset_created`、`dataset_exported`。
- Payout Ledger：按规则生成“待结算收益/积分”，不接真实支付。
- 审计日志：关键状态变更 append-only。
- 一个可演示的端到端 demo 数据集。

### 2.2 明确不做

- 真实邮件 MX/SMTP 收件。
- 浏览器插件、MCP、企业 SSO。
- 真实 OCR/ASR/视频理解。
- 真实 LLM 自动标注调用。
- 真实支付、发票、税务和提现。
- 公开数据交易市场。
- 多租户企业权限完整模型。
- 医疗、金融、未成年人等高敏数据商业化。
- “模型回答级归因”。
- 海外模型、海外日志、海外 SaaS 处理中国区数据。

## 3. 推荐技术栈

MVP 用“模块化单体 + 异步 worker”，工程上保持简单，但保留未来拆服务的边界。

| 层 | MVP 技术 | 选择理由 |
| --- | --- | --- |
| Frontend | Vite + React + TypeScript | 启动快，适合构建内部工作台和贡献者后台 |
| Backend API | FastAPI + Python | 适合文本处理、脱敏、解析、后续模型/规则扩展 |
| ORM/DB | SQLAlchemy + Alembic + PostgreSQL | 关系模型清晰，后续可加 pgvector |
| Queue | Redis + RQ 或 Dramatiq | 足够支撑异步解析、脱敏、标注、导出 |
| Object Store | 本地文件系统抽象，接口兼容 S3/OSS | 本地 MVP 简单，生产迁移到中国大陆 OSS/COS |
| Search/Dedup | Postgres FTS + hash/simhash | MVP 不引入独立向量库 |
| Export | 后端流式生成 JSONL/CSV/Markdown | 先验证数据产品交付 |
| Auth | Dev email login + session/JWT | 不引入复杂 OAuth |
| Deployment | Docker Compose | 一条命令跑通前后端、DB、Redis、Worker |

- 第一屏就是工作台，不做营销页。
- 所有原始内容不在普通页面展示，只展示脱敏内容。
- 处理状态必须清楚：uploaded、redacted、annotated、review_pending、approved、dataset_ready。
- 审核员页面必须展示“为什么进入审核”：隐私残留、重复、高价值、低置信、授权不足。
- 数据集详情必须展示：包含 Case 数、DRL 分布、重复率、人审比例、授权范围、导出记录。

## 9. 处理规则 MVP

### 9.1 脱敏规则

第一版使用规则引擎，不调用外部模型。

| 类型 | 示例 | 处理 |
| --- | --- | --- |
| 手机号 | `1xxxxxxxxxx` | `[PHONE_1]` |
| 邮箱 | `name@example.com` | `[EMAIL_1]` |
| 身份证样式 | 18 位身份证格式 | `[CN_ID_1]` |
| API key/token | `sk-...`, `AKIA...`, `ghp_...` | `[SECRET_1]` |
| URL query | `?token=...` | 删除 query 或 `[URL_WITH_QUERY]` |
| 地址样式 | 省市区街道门牌 | `[ADDRESS_1]` |
| 公司/客户名 | `有限公司`、`客户A` | `[ORG_1]`，低置信进入人审 |

必须做二次扫描：脱敏后的文本再次运行敏感规则；命中则停在 `privacy_review`。

### 9.2 标注规则

第一版通过关键词和结构特征标注：

- `software_engineering`：代码、报错、Git、API、数据库、部署。
- `customer_service`：客户、咨询、售后、投诉、工单。
- `sales_marketing`：线索、转化、文案、广告、私域。
- `legal_finance_hr`：合同、发票、报销、招聘、绩效。
- `agent_trace`：包含 tool call、terminal、browser、function result。

质量分由 5 个维度组成：

| 维度 | 权重 |
| --- | --- |
| 上下文完整度 | 25% |
| 任务清晰度 | 20% |
| 结果/反馈存在 | 20% |
| 可复用性 | 20% |
| 风险扣分 | -15% |

### 9.3 DRL 规则

| DRL | MVP 条件 |
| --- | --- |
| DRL0 | 原始隔离、脱敏失败、高敏拒收 |
| DRL1 | 脱敏通过，但标注不完整或授权仅限个人库 |
| DRL2 | 自动标注通过，基础授权通过，未人审 |
| DRL3 | 人审通过，授权覆盖商业数据集或评测用途 |

MVP 不生成 DRL4/DRL5，只预留字段。

## 10. 端到端演示脚本

验收时必须能按以下脚本跑通：

1. 启动系统：

```bash
cp .env.example .env
docker compose up --build
```

2. 打开前端：

```text
http://localhost:5173
```

3. 用开发态邮箱登录：`demo@lodia.local`。
4. 在 `/submit` 粘贴一段含邮箱、手机号、API key、AI 问答和用户反馈的样例。
5. 提交后进入 Case 详情，状态依次变化：

```text
uploaded -> quarantined -> parsed -> redacted -> residual_scanned -> normalized -> deduplicated -> annotated -> review_pending
```

6. 检查页面只展示脱敏内容，不展示原始手机号、邮箱或密钥。
7. 切换审核员角色，进入 `/review`。
8. 修改标签，审核通过，将 Case 升级为 DRL3。
9. 进入 `/datasets/new`，创建 `Demo Agent Eval Dataset`，选择 `min_drl=DRL3`。
10. 点击 Build，生成：

```text
manifest.json
quality_report.json
data.jsonl
data.csv
dataset.md
```

11. 下载 JSONL，确认每条记录包含：

```json
{
  "case_id": "...",
  "version_id": "...",
  "task_type": "...",
  "redacted_turns": [],
  "annotation": {},
  "license": {},
  "drl": "DRL3"
}
```

12. 进入 `/ledger`，确认产生：

```text
usage_event: dataset_exported
payout_event: pending
```

## 11. 里程碑

### M0：仓库脚手架和本地环境

- 新增 `apps/web`、`apps/api`、`apps/worker`。
- 新增 `docker-compose.yml`、`Makefile`、`.env.example`。
- Postgres、Redis、API、Worker、Web 一条命令启动。
- 健康检查：`GET /api/health`。

验收：`docker compose up --build` 后前端和 API 都可访问。

### M1：提交和隔离

- 实现开发态登录。
- 实现文本/文件提交。
- 写入 `submissions`、`raw_objects`。
- 原始内容保存到 `storage/raw`。
- 写入 `audit_logs`。

验收：提交样例后能看到 submission 状态和 raw object，不在普通 Case 页面展示原文。

### M2：Pipeline 第一版

- 实现解析、脱敏、残留扫描。
- 实现 Case Normalizer。
- 实现 hash 去重和轻量相似判断。
- 实现自动标注和 Quality Gate。
- 生成 DRL1/DRL2。

验收：样例能自动进入 `review_pending`，页面展示脱敏内容和自动标签。

### M3：人工审核工作台

- 实现审核队列。
- 实现通过、拒绝、改标签、设 DRL。
- 通过后生成 DRL3。
- 审核操作写入 `reviews` 和 `audit_logs`。

验收：审核员能把一个 Case 从 `review_pending` 推到 `approved_commercial`。

### M4：数据集生成和导出

- 实现 Dataset 创建。
- 实现 eligible case 查询。
- 实现 Data Contract MVP 检查。
- 生成 JSONL、CSV、Markdown、Manifest、Quality Report。
- 写入 `exports`。

验收：用户能下载一个只包含 DRL3 Case 的数据集。

### M5：Usage Ledger 和模拟收益

- 导出成功写 `usage_events`。
- 按简单规则生成 `payout_events`。
- 前端展示“预计收益/待结算”。

建议初始分账规则：

```text
base_value = 100 points
quality_multiplier = 0.5 + quality_score
novelty_multiplier = 0.5 + novelty_score
duplicate_penalty = 0.2 if duplicate else 1.0
payout_points = base_value * quality_multiplier * novelty_multiplier * duplicate_penalty
```

验收：数据集导出后贡献者账本出现 pending payout。

### M6：测试、样例和演示

- 固定 10 条 demo Case。
- 固定 PII/secret 脱敏回归样本。
- 固定重复样本回归测试。
- API 单测覆盖核心服务。
- Playwright 跑通浏览器端演示脚本。

验收：一条命令跑通 E2E：

```bash
make test
make e2e
```

## 12. 关键 ADR

### ADR-001：MVP 使用模块化单体

决策：API、Pipeline、Review、Dataset、Ledger 先在一个后端代码库内分模块实现，Worker 独立进程执行异步任务。

理由：当前目标是验证闭环，不是证明微服务治理能力。模块化单体可以降低部署、调试和数据一致性成本。

代价：后续吞吐上来后，需要按瓶颈拆服务。

### ADR-002：MVP 不调用真实 LLM

决策：脱敏和标注第一版全部使用规则、schema 和轻量相似度算法。

理由：避免数据出境、供应商合规、成本和模型不稳定问题；也能让 E2E demo 可重复。

代价：自动标注质量有限，必须通过人审进入 DRL3。

### ADR-003：原始数据只进 Raw Quarantine

决策：原始上传只能保存到 `storage/raw` 和 `raw_objects`，普通 Case API、审核台、导出服务都不能读取原文。

理由：这是 Lodia 信任体系的第一道边界。

代价：调试不如直接读原文方便，需要专门的管理员工具和审计。

### ADR-004：MVP 收益使用积分/待结算账本

决策：不接真实支付，只生成 `payout_events.status=pending` 和 points/amount preview。

理由：先验证“可追踪分账”而不是税务支付链路。

代价：商业上线前必须补齐实名、税务、打款、冻结和争议处理。

## 13. 风险和缓解

| 风险 | 影响 | MVP 缓解 |
| --- | --- | --- |
| 脱敏漏检 | 隐私风险 | 多规则扫描、二次扫描、命中高风险停留人审 |
| 过度脱敏 | 数据不可用 | 保留语义标签，如 `[ORG_1]`、`[ROLE_1]`、`[INDUSTRY_1]` |
| 自动标注不准 | 数据集质量低 | 自动标注只能 DRL2，人审后才 DRL3 |
| 重复刷量 | 收益失真 | canonical_hash + simhash + duplicate penalty |
| 授权不清 | 无法商用 | 提交时必须生成 consent snapshot，导出前检查用途 |
| Demo 太重 | 迟迟不能上线 | 第一版只支持文本/Markdown/JSON/TXT |

## 14. MVP 完成标准

MVP 完成不以功能多少衡量，而以闭环是否可信衡量。

必须满足：

- 用户能提交一个真实 AI 工作记录。
- 系统不在普通页面泄漏原始隐私。
- 系统能生成脱敏 Case、自动标签、质量分和 DRL。
- 审核员能把 Case 升级到 DRL3。
- 数据集导出只包含合格 Case。
- 每次导出有 Manifest、Quality Report、UsageEvent。
- 贡献者能看到该 Case 带来的待结算收益。
- 所有关键状态变更有 audit log。
- 全流程可以用 demo 数据在本地一条命令跑通。
